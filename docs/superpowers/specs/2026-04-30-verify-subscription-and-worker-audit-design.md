# Verify Subscription and Worker Audit Design

## Summary

This change set addresses two related goals without changing the legal business behavior, deployment shape, or operator workflow of the current pipeline:

1. Separate the subscription URL used by the verify stage from the subscription URL shown and deployed through the normal UI flow.
2. Refactor the Worker generation path so the emitted `_worker.js` remains functionally equivalent while becoming easier to audit, easier to maintain, and less likely to trigger false positives caused by legacy naming and stale template patterns.

The pipeline still produces:

- `artifact_dir/worker_transformed.js`
- `artifact_dir/_worker.js`
- `artifact_dir/pages_bundle/_worker.js`
- `artifact_dir/pages_bundle/modules/*.js`
- `artifact_dir/pages_bundle/manifest.json`

The Cloudflare Pages deployment command and upload structure stay unchanged.

## Goals

- Allow the verify stage to use its own configurable subscription URL.
- Default the verify subscription URL to `https://www.swimmingliu.xyz/sub?token=8410fb43eb2176497f5beafc0c39f5bc`.
- Keep the existing deploy subscription URL configurable and visible in the settings UI.
- Restore stable postprocess emoji decoration behavior for every final node.
- When country lookup fails, returns `ZZ`, or returns an invalid code, consistently fall back to `US` so the final grouping and emoji output do not appear blank or undecorated.
- Refactor Worker source generation with neutral, accurate naming and clear audit comments.
- Reduce false-positive risk caused by historical identifiers, stale comment text, and rigid template fingerprints.
- Preserve deployment behavior, request/response semantics, and artifact paths.

## Non-Goals

- Changing the Cloudflare deployment mechanism.
- Changing the secret query behavior or subscription payload format.
- Adding fake noise, misleading code, detection bypass logic, or review evasion behavior.
- Introducing a new bundler or replacing the current Python-driven Worker build path.

## Current Problems

### Verify URL coupling

The verify stage currently checks `deploy.subscription_url` directly. That couples:

- the subscription URL displayed to operators in the UI,
- the subscription URL stored for deployment-related settings,
- and the URL used for runtime verification.

This is too rigid when the verification target should be a different, explicitly managed endpoint.

### Emoji decoration regression

The postprocess stage still contains emoji decoration logic, but the resulting output can appear undecorated when country lookup fails or returns an invalid code. Today, invalid results normalize to `ZZ`, which maps to a white flag and can look like the emoji step disappeared.

### Worker auditability and false-positive risk

The Worker template and transform path still contain legacy identifiers and rigid patterns such as:

- `url_tag`
- `req_data`
- `MainData`
- `generated by vpn-subscription-automation (...)`
- direct string-fragment substitutions that are tied to the historical template shape

These do not accurately describe the Worker’s real role and make the emitted artifact look more template-derived than necessary.

## Design

### 1. Deploy configuration model

Extend `DeployConfig` with a dedicated `verify_subscription_url` field.

Rules:

- `subscription_url` remains the operator-facing and deployment-facing subscription URL.
- `verify_subscription_url` is used by the verify stage.
- If `verify_subscription_url` is blank, verification falls back to `subscription_url`.
- New defaults use:
  - `subscription_url`: current deploy default unless the user edits it
  - `verify_subscription_url`: `https://www.swimmingliu.xyz/sub?token=8410fb43eb2176497f5beafc0c39f5bc`

This keeps backward compatibility for older profiles while letting new profiles and UI state expose the verify-specific target explicitly.

### 2. Settings UI

The deploy settings drawer will expose four fields:

- project name
- Pages URL
- subscription URL
- verify subscription URL

Behavior:

- Project name auto-links the default Pages URL exactly as before.
- Editing Pages URL manually disables automatic overwrite exactly as before.
- Editing subscription URL continues to refresh the displayed QR code and subscription cards.
- Editing verify subscription URL affects only the verify stage, not the displayed QR code.

The settings help copy will explicitly explain that the verify URL is the health-check target used after deploy.

### 3. Verify stage

Both controller paths must use the same resolution logic:

- `effective_verify_subscription_url = deploy.verify_subscription_url or deploy.subscription_url`

That logic is used in:

- normal pipeline verify
- retry/resume verify

Verification output stays structurally the same:

- `secret_ok`
- `subscription_ok`

The only behavior change is which URL `subscription_ok` checks.

### 4. Postprocess fallback strategy

Postprocess will keep country lookup and emoji decoration, but invalid country results will no longer surface as `ZZ` in final grouping output.

Normalization policy:

- valid ISO-like two-letter country codes remain unchanged
- lookup failures, malformed values, and `ZZ` all degrade to `US`

Rendering policy:

- final decorated node names always use the normalized final code
- therefore invalid lookups become `🇺🇸 US <original name>`

This provides a stable operator-facing fallback and avoids the visual impression that the emoji stage was skipped.

### 5. Worker refactor scope

The refactor target is the Worker source generation path, not only the already-obfuscated artifact.

Primary files:

- `/Users/swimmingliu/data/VPN/vpn-subscription-automation/templates/vmess_node.js`
- `/Users/swimmingliu/data/VPN/vpn-subscription-automation/src/vpn_automation/pipeline/worker_build.py`
- related packaging/tests/docs

The emitted `_worker.js` remains the deployment artifact, but its source will come from a cleaner generation pipeline.

#### 5.1 Template responsibilities

The template will be rewritten with:

- neutral names that reflect behavior
- explicit helper boundaries
- concise audit-friendly comments

Expected responsibilities:

- read request URL and secret parameter
- choose between subscription payload and fallback random bytes
- encode the response with `btoa`
- return a `Response`

#### 5.2 Naming cleanup

Required renames:

- `MainData` -> `SUBSCRIPTION_PAYLOAD`
- `handleRequest` -> `handleSubscriptionRequest`
- `url_tag` -> `secretToken`
- `req_data` -> `responsePayload`
- `bytes` -> `randomBytes`
- `err` -> `error`

These names are part of the implementation contract and must stay:

- neutral
- accurate
- readable
- consistent across template and transform steps

#### 5.3 Build transform cleanup

`worker_build.py` will stop depending on brittle replacements tied only to old identifiers. Instead it will operate on a clearer, documented source shape.

Allowed improvements:

- constants consolidation
- helper extraction
- simpler manifest metadata
- accurate build banner text
- removal of legacy or misleading field names

Not allowed:

- fake padding
- deceptive comments
- dead branches inserted only to change fingerprints
- anti-analysis logic

### 6. Audit commentary in generated source

The transformed source should carry a short, accurate comment header describing the Worker at a high level, for example that it serves an encoded subscription payload when the configured secret query matches and otherwise returns random bytes.

The header must:

- help a reviewer understand purpose quickly
- not contain misleading claims
- not include stale product-history phrasing

### 7. False-positive risk reduction

False-positive reduction will come from normal engineering cleanup only:

- removing stale identifiers and test-like naming
- aligning names to actual behavior
- reducing rigid template fingerprints
- improving structure consistency between template, transformer, manifest, and deployed artifact

No evasion or concealment behavior will be added.

## Data Flow

### Verify URL flow

1. Profile load resolves `deploy.verify_subscription_url`.
2. Settings drawer edits and saves it.
3. Pipeline verify resolves the effective verify URL.
4. Cloudflare client verifies:
   - secret URL
   - effective verify subscription URL

### Postprocess flow

1. Country lookup resolves a raw code from host/IP.
2. Raw code is normalized.
3. Invalid or unknown codes degrade to `US`.
4. Final vmess name is decorated with `🇺🇸 US` or the resolved country.

### Worker build flow

1. Render stage injects final node payload into the template.
2. Worker build stage rewrites the rendered source into audit-friendly transformed source.
3. Obfuscation still consumes `worker_transformed.js` and emits `_worker.js`.
4. Packaging still writes `_worker.js`, sidecar modules, and manifest into `pages_bundle`.

## Testing Strategy

### Config and persistence

- verify `DeployConfig` supports `verify_subscription_url`
- verify TOML load/save round-trips the new field
- verify default profile contains the new verify URL

### UI

- verify deploy settings drawer renders the new field
- verify save flow persists both subscription URLs
- verify QR refresh continues to depend only on `subscription_url`

### Verify logic

- verify pipeline uses `verify_subscription_url` when present
- verify it falls back to `subscription_url` when blank
- verify retry/resume uses the same rule

### Postprocess

- verify valid country codes keep their own emoji
- verify `ZZ` degrades to `US`
- verify malformed codes degrade to `US`
- verify decorated node names always include emoji and final normalized country code

### Worker refactor

- verify transformed source contains the new neutral naming/header
- verify emitted Worker remains functionally equivalent for:
  - matching secret query
  - non-matching query
  - request error handling
- verify bundle outputs still exist in expected locations

### Full validation

- run unit tests
- run e2e tests
- run visual regression tests
- run a UI verification pass for the settings drawer and results/subscriptions pages
- repackage Electron after merge-ready validation

## Risks and Mitigations

### Risk: backward compatibility for existing profiles

Mitigation:

- default `verify_subscription_url` during deserialization when missing
- fall back to `subscription_url` during verify execution

### Risk: UI confusion between two subscription URLs

Mitigation:

- clear field labels
- help text explaining display/deploy URL versus verify URL

### Risk: Worker refactor changes runtime behavior

Mitigation:

- treat request/response behavior as locked contract
- verify with focused unit tests before broad regression runs

## Deliverables

- updated config model and persisted profile support for `verify_subscription_url`
- settings UI support for custom verify subscription URL
- postprocess fallback behavior that normalizes invalid country results to `US`
- refactored Worker generation path producing audited `_worker.js`
- concise refactor notes that separate:
  - code quality improvements
  - false-positive risk reductions
  - structural consistency changes
- functional equivalence verification results
